## CRITICAL

### [CHOLA_MS] CHOLA_MS-1 — MERGED_CELL_DROP
**Location:** `extract_chola.py:195 (TW loop `for r0 in range(6, 156, 6)`); 'TW June 26' sheet`  
**Problem:** The TW loop assumes every state block is exactly 6 rows apart (fixed step 6 starting at row 6). This holds only for the first 9 states (AP@6 .. GA@54). The GA block is 6 rows tall (B54:B59) because it contains an extra 'SOD' row, shifting every subsequent state block by +1. From BR (row 61) onward the step-6 indices (60,66,72,...) all land on blank state cells, so the `if not (state...)` guard skips them. 16 entire state blocks are silently dropped: BR, JH, CG, OD, WB, AS/ML/TR/AR/NL, DL, HR, HP, JK/LD, PB, MP, RJ, UK, East UP, AN.  
**Fix:** Drive the TW loop off the actual merged-cell ranges in column B (ws.merged_cells.ranges where min_col==2 and min_row>=6) instead of a hardcoded range(6,156,6). Iterate each block's true start row and true height; do not assume a uniform 6-row stride.

### [CHOLA_MS] CHOLA_MS-2 — VALUE_FIDELITY
**Location:** `extract_chola.py:102-105 (TW_TYPES offsets) applied to GA block rows 54-59`  
**Problem:** The GA block has 6 rows because row 57 is a 'SOD' (SAOD) row inserted between the NEW slabs and ANNUAL: r54 NEW(100-500), r55 NEW(30-100), r56 NEW(UPTO30), r57 SOD, r58 ANNUAL, r59 ACT. The extractor's fixed TW_TYPES maps offsets 0-4 onto rows 54-58, so it (a) labels row 57's SOD values as 'COMP-RENEWAL', (b) labels row 58's ANNUAL/renewal values as 'SATP', and (c) never reads row 59 (the real ACT/SATP). GA's renewal and SATP payouts are therefore wrong and GA's true SATP row is dropped.  
**Fix:** Read the per-row Type label from column C (and the EV mirror column AQ) and map by label (NEW slabs / ANNUAL / SOD / ACT) instead of by fixed offset, so blocks with an extra SOD row are handled and SATP comes from the true ACT row.

### [GODIGIT] GODIGIT-1 — SCHEMA_PIPELINE
**Location:** `extract_godigit.py:161-235 (CV Grid loop), CV Grid (excl. HCV) header row 230 + data rows 231-260`  
**Problem:** The CV Grid loop hardcodes Segment=col B (ws.cell(r,2)) and Make=col C (ws.cell(r,3)). But several blocks have an EXTRA leading 'Cluster' column, so their header is Cluster|Segment|Make|... (col B=Cluster, col C=Segment, col D=Make). For these blocks the extractor reads the CLUSTER name as the segment; _cat() then returns '' (e.g. _cat('Patna')) and the ENTIRE block is dropped. The Taxi blocks for Patna / Lucknow / Bhubuneshwar / Bihar / Orissa / Good UP are completely absent from the catalog.  
**Fix:** Detect the leading-Cluster layout at the header row (when a 'Cluster' token precedes 'Segment'): build a column map for Segment/Make/Cluster from the header tokens instead of hardcoding cols B/C. Use the per-row Cluster col as geo_label for these blocks.

### [RESOLVER] F01-DECLINE-NOT-ENFORCED — RESOLVER_LOGIC
**Location:** `api/_engine.py:432-449 (quote_catalog), specifically the fallback at 439-442`  
**Problem:** A FIRED decline does NOT override an available rate. quote_catalog computes rate candidates independently of declines, and only returns decision=DECLINE when there are zero rate candidates (`'DECLINE' if declines else 'NO_RATE'` is reached only inside `if not cands`). When any RATE row matches (e.g. a blank-make PAN_INDIA wildcard), the decline is demoted to an informational `declines` field and the engine returns ALLOW with a pay-in %.  
**Fix:** In quote_catalog, if `declines` is non-empty and a matched decline binds the same risk, return DECLINE (or strip/zero the rate) BEFORE selecting a rate candidate, mirroring quote()/check_eligibility ordering where declines short-circuit. At minimum, a decline whose specificity >= the winning rate's make-specificity must win.

### [RESOLVER] F02-EXCEPT-MAKE-INVERTED-RATE — MERGED_CELL_DROP
**Location:** `api/_engine.py:308-328 (_make_ok); also _tier_names 285-291`  
**Problem:** _make_ok handles only 'OTHER THAN' and 'NON-'/'NON ' negations. It has NO handling for 'EXCEPT'/'EXCLUDING'/'EXCL.' phrasing. For labels like 'All (except Tata, Mahindra and AL)' and 'All excluding Volvo and Scania*', it falls through to `_loose(rm, tm)` substring matching, which INVERTS the rule: the excluded make matches (its name is a substring of the label) and every make that SHOULD be covered fails.  
**Fix:** In _make_ok add negation parsing for 'EXCEPT'/'EXCLUDING'/'EXCL.': strip leading 'ALL', extract the excluded list after the keyword, split on &/,/ 'and', and return (risk_make NOT in excluded_list, 1). Mirror the same in _decline_match (F03).

### [RESOLVER] F04-POLICY-TYPE-EXACT-MATCH — VALUE_FIDELITY
**Location:** `api/_engine.py:337-339 (_rate_match policy_type) and 375-378 (_decline_match policy_type)`  
**Problem:** policy_type is compared by exact equality (`r.get('policy_type') != risk['policy_type']`), but the grids store compound/qualified labels: SBI 'COMP & SATP', 'COMP (PACKAGE)', 'COMP-NIL DEP'; CHOLA 'COMP-NEW (100-500 NOP)', 'COMP-RENEWAL'; GODIGIT 'COMP (1+5)', 'COMP (5+5)', 'COMP-NB'; TATA_AIG only 'COMP-RENEWAL'/'COMP-ROLLOVER'/'COMP-BRANDNEW'; ICICI 'COMP-NB (1+3/3+3)', 'SAOD/COMP [NON-NCB RENEWAL]'. A normal query policy_type='COMP' or 'SATP' fails to match any of these.  
**Fix:** Normalize/prefix-match policy_type (e.g. treat 'COMP*' family as COMP, 'SAOD*' as SAOD, 'COMP & SATP' as matching both COMP and SATP), or expose the exact internal label as a required dimension. Substring/family matching consistent with how make is matched is the minimal fix.

### [GEO] GEO-01 — RESOLVER_LOGIC
**Location:** `geo_normalize.py:96-125 (to_state, step 4 embedded-state vs step 7 composite)`  
**Problem:** Composite labels whose parts disagree on state are silently resolved to ONE wrong state instead of None. The embedded-state branch (step 4) runs BEFORE the composite-agreement branch (step 7), so as soon as one state name is embedded in the string the function returns it and never reaches the 'all parts must agree' guard.  
**Fix:** Run the composite split (step 7) FIRST when the label contains ',' '&' or '/', and only fall back to the single embedded-state scan when the split yields 0 or 1 resolvable parts. Require unanimous agreement among resolvable parts before returning a state.

### [GODIGIT] GODIGIT-4 — COVERAGE_GAP

### [HDFC_ERGO] HDFC_ERGO-1 — BAND_BOUNDARY
**Location:** `extract_misc_4w.py extract_hdfc() lines 165-185; grid note Grid!H6:H7`  
**Problem:** The Grid footnote defines an explicit engine-cc band that the extractor does not encode. H6: 'The BDE% mentioned in the grid is for 1000 and above CC'. H7: 'For 0 - 1000 CC, BDE will be 5% less than the mentioned grid'. The extractor emits each of the 58 rates with cc_min/cc_max BLANK (unbounded), so the resolver applies the 1000cc+ rate to ALL private cars, including sub-1000cc vehicles, over-stating commission by 5 percentage points for the entire 0-1000cc population (e.g. Mumbai Petrol shows 60% but 0-1000cc should be 55%; WB CNG 25% should be 20%). The catalog schema already has cc_min/cc_max fields available to express this, and the extractor could have set cc_min=1000 on the listed rows and emitted a parallel 0-1000cc row at (rate-5). It is currently only flagged as a PENDING warning, not extracted, so wrong rates ship.  
**Fix:** For each Grid data row emit TWO atomic rows: one with cc_min=1000, cc_max='' at the listed BDE%, and one with cc_min=0, cc_max=999 (or <1000 boundary) at pct(BDE)-5. Keep the warning until values are confirmed with the insurer, but stop shipping a single cc-unbounded rate that is wrong for sub-1000cc cars.

### [PIPELINE] F2 — SCHEMA_PIPELINE
**Location:** `api/_engine.py:36-39 (_RTO2CLUSTERS) + refresh() lines 523-534; api/_store.py put_grid (104-114)`  
**Problem:** RTO->cluster geo maps (_RTO2CLUSTERS) are built once at import from clusters.json and are NOT rebuilt in refresh() (only _CLUSTER2STATES is referenced there, and even that is not reassigned — refresh rebinds only CATALOG/OPTIONS/_TIER_IDX/STATES_ALL). put_grid stores only extracted rule rows, never cluster maps. So when a CHOLA_MS or GODIGIT grid is re-uploaded with changed RTO-cluster vocabulary, _geo_ok resolves rto_code against the STALE baked cluster map. An RTO code newly present in the upload but absent from the baked map hits the `if not labels: return False` branch (engine line 252) and silently fails to match / declines.  
**Fix:** Either persist per-insurer cluster maps alongside the uploaded grid and rebuild _RTO2CLUSTERS/_CLUSTER2STATES in refresh(), or document that uploads cannot change RTO-cluster geography.

### [PIPELINE] F3 — OTHER
**Location:** `extract_chola.py:179-181 nid(); extract_godigit.py:128-130 nid(); extract_misc_4w.py:40-42 _mk(); review keyed by catalog_id in api/_store.py + api/index.py /review`  
**Problem:** catalog_id is a positional sequence counter per (insurer, kind): CHOLA-TW-0001, CHOLA-TW-0002, ... assigned by enumeration order during extraction, NOT derived from cell/content. Review decisions are stored in KV keyed by catalog_id (store.upsert / GRID review:catalog hash). When a grid is re-uploaded, rows are renumbered by position; if the source layout shifts (a row inserted/removed/reordered), an existing CONFIRMED/REJECTED review on e.g. CHOLA-TW-0399 silently re-binds to a DIFFERENT rule. apply_reviews (engine 554) joins purely on catalog_id, so a stale approval can carry a wrong pay_in_pct edit onto an unrelated rule. source_rule_id holds the stable cell ref (e.g. Sheet1!O12) and is populated on 100% of rows, but is unused for review keying.  
**Fix:** Key review state on a content hash or source_rule_id (cell+scope), or invalidate/migrate reviews for an insurer when its grid is re-uploaded.

### [RESOLVER] F03-EXCEPT-MAKE-INVERTED-DECLINE — RESOLVER_LOGIC
**Location:** `api/_engine.py:398-410 (_decline_match make/model block, line 400 `_loose(risk['make'], r['make'])`)`  
**Problem:** _decline_match uses plain `_loose` substring matching for the decline make with no EXCEPT/EXCLUDING handling, so 'All excl. Bajaj and TVS' style decline rows fire INVERTED: they decline the very makes that are excluded from the decline (i.e. the allowed ones) and fail to decline everyone else.  
**Fix:** Add EXCEPT/EXCLUDING/EXCL. negation parsing to _decline_match identical to the _make_ok fix; for 'All excl X' fire the decline iff risk make is NOT in X.

### [RESOLVER] F05-INSURER-ISOLATION-LEAK — RESOLVER_LOGIC
**Location:** `api/_engine.py:422-431 (quote_catalog rules/declines filter)`  
**Problem:** When no `insurer` is supplied, quote_catalog pools RATE and DECLINE rows across ALL insurers in the category. One insurer's PAN-India sourcing ban (a CAT_B 'effect=DECLINE', PAN_INDIA, no make/policy) then appears as a decline for queries that are actually rate-eligible under a different insurer, and (combined with F01) can DECLINE a query that another insurer would serve when no rate matches.  
**Fix:** Only evaluate declines belonging to the same insurer as the candidate rate being considered, or require `insurer` for any DECLINE evaluation; never let insurer A's decline appear against insurer B's quote.

### [RESOLVER] F06-GEO-DECLINE-FIRES-WHEN-GEO-OMITTED — GEO_MAPPING
**Location:** `api/_engine.py:273-275 (_geo_ok 'no geo supplied at all -> unconstrained' -> returns True,0), invoked from _decline_match line 379`  
**Problem:** _geo_ok returns (True,0) when the risk supplies no state/cluster/rto. For a geo-scoped DECLINE (e.g. Bihar-only) this means the decline FIRES for any query that omits geo. Declines should require the bound geo to be present and matching, not treat absence as a wildcard hit.  
**Fix:** In the decline path, treat a rule that binds a geo as non-matching when the risk provides no geo (return False), rather than relying on _geo_ok's unconstrained wildcard. Keep the wildcard behavior only for rate fallback.

### [RESOLVER] F08-RANKING-TIE-NONDETERMINISTIC — RESOLVER_LOGIC
**Location:** `api/_engine.py:443-444 (cands.sort(key=lambda t: t[0], reverse=True); best=cands[0][1])`  
**Problem:** Candidates are sorted by specificity score only; ties are broken by Python's stable sort (i.e. original catalog order), not by any business rule (e.g. lowest/most-conservative pay-in, or NOP/age sub-band). When several rows tie on score with DIFFERENT pay_in_pct, the winner is effectively arbitrary. This is common because policy sub-bands (NOP count, NB vs renewal) and other distinctions live inside non-modeled fields.  
**Fix:** Add a deterministic, business-meaningful tie-break (e.g. prefer the more specific make tier, then a documented conservative pay-in rule) and surface ambiguous ties for review instead of silently picking the first.

### [GEO] GEO-08 — BAND_BOUNDARY
